GRIP — PHASE 2 백엔드 테스트 계획
작성일: 2026-05-24

테스트 도구: supertest + node:test
테스트 위치: tests/backend/
실행 명령: npm run test:backend

전제 조건:
  - .env에 실제 Supabase 연결 정보 설정
  - HMAC_SECRET 설정
  - MAX_DISTANCE_METERS=100
  - DISABLE_AI=true (AI 테스트는 별도 처리)
  - 테스트 계정 3개 사전 등록 (merchant / consumer / admin)

================================================================================
1. 공통 응답 포맷 규칙 (모든 케이스에 적용)
================================================================================

성공 응답:  { success: true,  data: { ... } }
실패 응답:  { success: false, error: { code: string, message: string } }

HTTP 상태코드 매핑:
  200  정상 응답
  201  리소스 생성 성공
  400  입력 오류 / QR 위변조 / 잔액 부족
  401  인증 실패 / SQLi 탐지
  403  계정 잠금 / 권한 없음 / 차단된 계정
  404  리소스 없음
  429  Rate limit 초과
  500  서버 내부 오류

2-2. POST /api/auth/login
────────────────────────────────────────

[ TC-L01 ] 정상 로그인
  입력: { email: "consumer@test.com", password: "password1" }
  기대: 200, data.token (JWT 문자열), data.user.role="consumer"

[ TC-L02 ] 잘못된 비밀번호
  입력: { email: "consumer@test.com", password: "wrongpassword" }
  기대: 401, error.code="INVALID_CREDENTIALS"

[ TC-L03 ] 존재하지 않는 이메일
  입력: { email: "nobody@test.com", password: "password1" }
  기대: 401, error.code="INVALID_CREDENTIALS"
  ※ 이메일 열거 방지: 존재 여부를 응답으로 구분 불가해야 함

[ TC-L04 ] SQLi 탐지 — email 필드
  입력: { email: "' OR '1'='1", password: "password1" }
  기대: 401, error.code="SQLI_DETECTED"
  검증: security_events에 SQLI_BLOCKED 이벤트 기록 확인

[ TC-L05 ] SQLi 탐지 — password 필드
  입력: { email: "test@test.com", password: "'; DROP TABLE users;--" }
  기대: 401, error.code="SQLI_DETECTED"
  검증: security_events에 SQLI_BLOCKED 이벤트 기록 확인

[ TC-L06 ] SQLi 탐지 패턴 목록 (각각 개별 테스트)
  패턴: ' / " / ; / -- / # / /* */ / UNION / SELECT / DROP / INSERT / UPDATE
  기대: 모두 401, error.code="SQLI_DETECTED"

[ TC-L07 ] Rate limit 초과 (분당 11회)
  입력: 동일 IP에서 POST /api/auth/login 11회 연속 호출
  기대: 1~10회는 통과, 11회째는 429, error.code="RATE_LIMITED"
  검증: security_events에 BRUTE_FORCE 이벤트 기록 확인

[ TC-L08 ] 계정 잠금 (5회 연속 실패)
  입력: 동일 계정에 잘못된 비밀번호 5회
  기대:
    1~4회: 401, error.code="INVALID_CREDENTIALS"
    5회:   403, error.code="ACCOUNT_LOCKED", error.unlock_at (ISO 타임스탬프)
  검증: unlock_at이 현재 시각 + 약 30분임을 확인

[ TC-L09 ] 잠긴 계정으로 즉시 재로그인
  선행: TC-L08 완료
  입력: 올바른 비밀번호로 로그인 시도
  기대: 403, error.code="ACCOUNT_LOCKED", unlock_at 포함

  케이스:
    동일 좌표 → 0m
    대전(36.3504, 127.3845) ↔ 서울(37.5665, 126.9780) → 약 148,000m ± 1%
    100m 간격 좌표 → 약 100m ± 1m

────────────────────────────────────────
8-2. Rate Limit + 계정 잠금 연동
────────────────────────────────────────

[ TC-RL01 ] Rate limit 해제 후 재접근
  선행: Rate limit 발생 (11회 요청)
  대기: 60초 경과 후 재요청
  기대: 429 아닌 응답 (401 또는 200)

[ TC-RL02 ] Rate limit은 IP 단위임을 확인
  선행: IP-A에서 rate limit 발생
  요청: IP-B에서 로그인 시도
  기대: 429 아닌 응답

[ TC-RL03 ] Rate limit과 계정 잠금 순서
  동작: Rate limit 상태에서도 계정 잠금 확인 요청
  기대: Rate limit이 계정 잠금보다 먼저 체크됨 (429 반환)

────────────────────────────────────────
8-3. 차단 계정 전체 흐름
────────────────────────────────────────

[ TC-BL01 ] 차단된 계정 로그인 시도
  선행: consumer 차단
  기대: 403, error.code="USER_BLOCKED", message=block_reason

[ TC-BL02 ] 차단된 계정 결제 시도
  선행: consumer 차단 + 유효한 QR 존재
  기대: 403, error.code="USER_BLOCKED"

[ TC-BL03 ] 차단 해제 후 정상 로그인 가능
  선행: TC-BL01 → 차단 해제 → 로그인 재시도
  기대: 200, data.token 반환
