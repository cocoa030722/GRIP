전통시장 QR 결제 시스템

기존 QR 결제 시스템의 위험:
QR 스티커 덮어치기(QRLJacking, Quishing) — 공격자가 자기 QR을 위에 덮어붙임. 손님 돈이 공격자 계좌로
QR 위변조 — 정상 QR을 캡처해 가짜 사이트로 유도
금액 변조 — 동적 QR에서 결제 금액을 중간에서 바꿈

핵심 동작 흐름 (동적 서명 QR):
상인이 앱에서 금액 입력 → 결제 서버에 "결제 요청" 생성
서버가 {payment_id, merchant_id, amount, exp, nonce} + HMAC 서명 을 QR로 발급, 상점에서 새 qr을 생성하면 과거 qr은 무효
소비자가 스캔 → 앱이 서버에 검증 요청
서버가 HMAC 검증 + 유효기간 + 중복사용(nonce) 확인 → 통과 시 결제 진행
거래 로그를 해시 체인으로 DB에 적재 (위변조 탐지)

구현 시 참고사항: 실제 결제 시스템은 구현하지 않음. 웹페이지 내부의 포인트를 현금으로 간주하고 구현

① SQL Injection (심사표 명시)
공격: 로그인 폼에 ' OR '1'='1 입력
방어: Prepared statement 또는 ORM (SQLAlchemy/Prisma) 사용
시연: Burp Suite 또는 그냥 브라우저에서 페이로드 입력 → 로그인 거부 + security_events 테이블에 SQLI_BLOCKED 기록 → 대시보드에 알림 뜨는 거 라이브로

② Brute Force (심사표 명시)

공격: 간단한 Python 스크립트로 1초에 50회 로그인 시도
방어: Redis 기반 rate limiting (IP당 분당 10회) + 5회 실패 시 30분 계정 잠금
시연: 공격 스크립트 실행 → 6회째부터 차단 + 대시보드에 BRUTE_FORCE 이벤트 폭주

③ QR 위변조 / 덮어치기 (이 대회의 차별화 포인트)

공격: 정상 QR을 캡처해서 가짜 QR 생성 (서명 없거나 다른 가게 ID)
방어: HMAC 서명 검증 + nonce 1회용 
시연: 가짜 QR 스캔 → "위변조된 QR입니다" + 정상 QR도 두 번 스캔하면 두 번째 거부 (replay 방지)

④ 거래 데이터 무결성 (해시 체인)
공격: DB에 직접 들어가서 거래 금액 5,000원 → 50,000원으로 변조
방어: 거래 시 current_hash = SHA256(prev_hash + tx_data)로 체인 구성. 검증 시 전체 체인 재계산
시연: 실제로 DB 한 줄 UPDATE → 무결성 검증 API 호출 → "X번째 거래에서 체인 깨짐" 탐지

대시보드에 뭘 보여줄지 (발표 임팩트 설계)
심사위원이 보는 화면이 점수의 30점(발표/협업)을 좌우합니다. 다음 4개 컴포넌트를 우선순위 순으로 구현하세요.
① 실시간 이벤트 피드 (필수)
[17:23:01] 🔴 BRUTE_FORCE    192.168.1.45  → 로그인 11회 차단
[17:23:04] 🟡 SQLI_BLOCKED   203.0.113.7   → /api/login 인젝션 탐지
[17:23:09] 🔴 INVALID_QR     -             → 위변조 QR 스캔 시도
[17:23:15] 🟢 PAYMENT_OK     -             → ₩12,000 정상 결제
공격 시연하면 이 피드가 실시간으로 흘러내려가는 게 발표장 분위기를 압도합니다.
② 이벤트 타입별 막대 차트 (필수)
Chart.js나 Recharts로 1분 단위 집계. 공격 스크립트 돌리는 순간 막대가 솟아오르는 걸 보여주세요.
③ 결제 성공/실패 비율 (있으면 좋음)
도넛 차트. 정상 결제 vs 차단된 시도.
④ 대전 시장 지도 + 결제 히트맵 (있으면 좋음)
공공데이터 위경도 + Leaflet. 결제 발생 시 마커가 반짝이면 최고인데, 시간 부족하면 스킵.
