# GRIP 기획안
**전통시장 QR 결제 보안 시스템**

---

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| **프로젝트명** | GRIP (Guard + Real-time + Integrity + Payment) |
| **발표 행사** | SOGRA 해커톤 |
| **한 줄 요약** | 전통시장 QR 결제의 위변조·SQL Injection·Brute Force를 실시간 탐지·차단하는 보안 데모 시스템 |

---

## 2. 문제 정의

전통시장의 QR 결제 확산과 함께 **3가지 핵심 보안 위협**이 부각되고 있습니다.

| # | 위협 | 설명 |
|---|------|------|
| 1 | **QR 위변조** | 악의적 QR로 상인을 사칭하거나, 이미 사용된 QR을 재사용해 이중 결제 유도 |
| 2 | **SQL Injection** | 로그인 폼 입력값으로 DB를 직접 조작, 인증 우회 시도 |
| 3 | **Brute Force** | 자동화 스크립트로 비밀번호 반복 시도, 계정 탈취 |

> 소규모 시장 상인은 별도 보안 인프라를 갖추기 어렵고, 기존 QR 결제 솔루션은 위 공격에 대한 방어 로직이 부재합니다.

---

## 3. 솔루션: GRIP

GRIP은 **결정론적 룰 기반 방어** + **AI 이상 행동 분석**을 결합해 전통시장 결제 환경에 특화된 3중 보안 레이어를 제공합니다.

```
[결제 요청]
    │
    ├─ 1. QR 서명 검증 (HMAC-SHA256 + GPS + nonce)
    ├─ 2. SQL Injection 탐지 (미들웨어 레이어)
    ├─ 3. Brute Force 차단 (슬라이딩 윈도우 + 계정 잠금)
    │
    └─ [통과 시] 포인트 이체 완료
              │
              └─ AI 이상 패턴 분석 → 관리자 권고 (비동기)
```

---

## 4. 핵심 기능

### 4-1. 동적 서명 QR 결제 (위변조 방어)

- 상인이 금액 입력 + GPS 위치를 포함해 **HMAC-SHA256 서명 QR** 발급
- 소비자 스캔 시 서버가 **3단계 검증** 수행

| 검증 단계 | 차단 코드 | 설명 |
|-----------|-----------|------|
| 서명 검증 | `INVALID_QR` | 서명 불일치 또는 만료 QR |
| nonce 검사 | `REPLAY_QR` | 이미 사용된 QR 재사용 |
| 거리 계산 | `LOCATION_MISMATCH` | 상인·소비자 간 100m 초과 |

### 4-2. SQL Injection 방어

- 로그인 요청의 이메일·비밀번호 필드를 **미들웨어 레이어**에서 선차단
- DB 쿼리 실행 전 차단 → Parameterized Query와 **2중 방어**
- 탐지 즉시 `SQLI_BLOCKED` 이벤트 기록

```
패턴: /('|"|;|--|#|\/\*|\*\/|UNION|SELECT|DROP|INSERT|UPDATE)/i
```

### 4-3. Brute Force 방어

- **IP 기반 슬라이딩 윈도우** (60초 / IP당 최대 10회, 외부 의존성 없음)
- 10회 초과 시 즉시 **429** 반환 + `BRUTE_FORCE` 이벤트 기록
- 동일 계정 5회 연속 실패 시 **30분 계정 잠금** (`ACCOUNT_LOCKED`)

### 4-4. AI 이상 행동 탐지

- **Ollama 로컬 LLM (Gemma 4)** 이 트랜잭션·이벤트 기록을 주기적으로 분석
- 패턴 요약·자연어 차단 사유 생성 → 관리자에게 차단 **권고**
- AI는 설명 보조 역할; 실제 차단은 결정론적 룰이 담당

### 4-5. 실시간 보안 이벤트 대시보드

- **SSE(Server-Sent Events)** 로 공격 발생 즉시 화면 반영 (새로고침 불필요)
- Chart.js 막대 차트로 이벤트 통계 시각화
- 이벤트 타입별 필터 (SQLI / 브루트포스 / 결제 / 전체)

## 5. 시스템 아키텍처

```
┌─────────────────────────────────────────────────────┐
│                   Browser (Vanilla JS)               │
│  index.html  payment.html  merchant.html  dashboard.html │
└──────────────────────┬──────────────────────────────┘
                       │ HTTP / SSE
┌──────────────────────▼──────────────────────────────┐
│               Node.js + Express (API Server)         │
│                                                      │
│  Middleware:                                         │
│    sqliDetect.middleware.js  ← SQL Injection 차단    │
│    rateLimit.middleware.js   ← Brute Force 차단      │
│    auth.middleware.js        ← JWT 검증              │
│                                                      │
│  Core Libs:                                          │
│    lib/hmac.js       ← QR 서명/검증                  │
│    lib/haversine.js  ← GPS 거리 계산                 │
│    lib/localAI.js    ← Ollama LLM 호출               │
└──────────────────────┬──────────────────────────────┘
                       │ Supabase Client
┌──────────────────────▼──────────────────────────────┐
│           Supabase (PostgreSQL + Auth)               │
│  users / qr_codes / transactions / security_events  │
└─────────────────────────────────────────────────────┘
                       │ (비동기)
┌──────────────────────▼──────────────────────────────┐
│          Ollama Local LLM (Gemma 4)                  │
│  트랜잭션/이벤트 패턴 분석 → 자연어 권고 생성        │
└─────────────────────────────────────────────────────┘
```

---

## 6. 기술 스택

| 영역 | 기술 |
|------|------|
| **백엔드** | Node.js + Express |
| **DB / Auth** | Supabase (PostgreSQL + Auth) |
| **프론트엔드** | Vanilla JS, Chart.js (CDN) |
| **보안 암호** | Node.js 내장 `crypto` (HMAC-SHA256) |
| **Rate Limiting** | In-memory 슬라이딩 윈도우 (외부 의존성 없음) |
| **AI 분석** | Ollama 로컬 LLM + Gemma 4 |
| **실시간 통신** | SSE (Server-Sent Events) |

## 7. 데모 시나리오 (해커톤 발표)

### 시나리오 1 — 정상 결제 흐름
1. 상인: `merchant.html`에서 5,000원 QR 발급 (GPS 포함 서명)
2. 소비자: `payment.html`에서 QR 스캔 → 결제
3. 대시보드: `PAYMENT_OK` 이벤트 실시간 표시

### 시나리오 2 — SQL Injection 탐지
1. 로그인 폼에 `' OR '1'='1` 입력 → 로그인 클릭
2. 화면: "SQL Injection 패턴이 감지되었습니다" 에러
3. 대시보드: `SQLI_BLOCKED` 카드 실시간 등장

### 시나리오 3 — Brute Force 탐지
1. `attacks.py` 실행 → 30회 연속 로그인 시도
2. 터미널: 11번째 요청부터 `429` 반환
3. 대시보드: `BRUTE_FORCE` 카드 등장 + 차트 막대 증가
4. (추가) 5회 실패 계정 → `ACCOUNT_LOCKED` 확인

## 8. 기대 효과

| 관점 | 효과 |
|------|------|
| **소비자** | 위변조 QR 결제 사기 차단, 안전한 결제 경험 |
| **상인** | 사칭 QR로 인한 피해 방지, 포인트 정상 정산 |
| **운영자** | 공격 시도 실시간 모니터링, AI 기반 이상 패턴 권고 |
| **기술** | 외부 보안 인프라 없이 순수 Node.js + In-memory 로 구현 가능한 경량 보안 |

---

## 9. 차별점

1. **QR에 GPS 좌표 포함** — 물리적 위치 검증으로 원격 QR 도용 원천 차단
2. **미들웨어 선차단** — DB 도달 전 SQL Injection 방어, Parameterized Query와 2중 구조
3. **AI는 보조만** — 결정론적 룰이 차단 주체, AI는 설명·권고 생성으로 역할 분리
4. **외부 의존성 최소화** — Rate limiting을 In-memory로 구현, 전통시장 소규모 서버 환경에 적합
