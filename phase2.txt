────────────────────────────────────────
1-A. lib/hmac.js
────────────────────────────────────────
목적: QR 서명 생성 및 검증

export:
  sign(paymentId, merchantId, amount, nonce, lat, lng) → string (hex)
  verify(paymentId, merchantId, amount, nonce, lat, lng, signature) → boolean

서명 입력 포맷 (spec 4.1):
  message = `${paymentId}:${merchantId}:${amount}:${nonce}:${lat.toFixed(4)}:${lng.toFixed(4)}`
  키 = process.env.HMAC_SECRET
  알고리즘 = crypto.createHmac('sha256', key).update(message).digest('hex')

verify는 timingSafeEqual로 비교 (타이밍 공격 방지)

────────────────────────────────────────
1-B. lib/haversine.js
────────────────────────────────────────
목적: 두 GPS 좌표 간 거리 계산 (미터)

export:
  distance(lat1, lng1, lat2, lng2) → number (미터, DOUBLE)

공식 (spec 4.4):
  R = 6371000
  Δlat = (lat2 - lat1) * π/180
  Δlng = (lng2 - lng1) * π/180
  a = sin²(Δlat/2) + cos(lat1*π/180) * cos(lat2*π/180) * sin²(Δlng/2)
  distance = 2 * R * Math.asin(Math.sqrt(a))

2-A. controllers/payments.controller.js
────────────────────────────────────────

함수 3개: requestQR / verifyQR / getHistory

──── requestQR (POST /api/payments/request) ────

미들웨어: authMiddle (req.user 주입됨)
역할 확인: req.user.role === 'merchant' 아니면 403 FORBIDDEN

입력 (body): amount (정수 > 0), merchant_lat, merchant_lng
  - 셋 중 하나라도 누락 → 400 LOCATION_REQUIRED 또는 INVALID_INPUT
  - amount <= 0 → 400 INVALID_INPUT

처리 순서:
  1. merchant_id = req.user.id
  2. 기존 pending QR 일괄 만료:
       supabaseAdmin.from('merchant_qr_codes')
         .update({ status: 'expired', revoked_at: now(), revoked_reason: '상인 QR 갱신' })
         .eq('merchant_id', merchant_id).eq('status', 'active')
  3. nonce = crypto.randomUUID()
  4. hmac_signature = hmac.sign(qr_id예약불가→ 선삽입 방식 사용)
     → 먼저 merchant_qr_codes INSERT (hmac_signature = '' 임시)
     → qr_id 획득 후 서명 계산 → UPDATE hmac_signature
     ※ 또는: qr_id = crypto.randomUUID() 직접 생성 후 한 번에 INSERT
       (gen_random_uuid() 대신 서버에서 생성하면 서명에 포함 가능)
     구현 선택: 서버에서 qr_id = crypto.randomUUID() 생성 → sign(qr_id, ...) → INSERT
  5. merchant_qr_codes INSERT:
       { qr_id, merchant_id, merchant_lat, merchant_lng,
         hmac_signature, expires_at: +24h (더미, status로만 관리),
         status: 'active' }
  6. qr_payload = `${qr_id}:${hmac_signature}`
  7. 응답 201:
       { success: true, data: { qr_id, hmac_signature, qr_payload, expires_at } }

──── verifyQR (POST /api/payments/verify) ────

미들웨어: authMiddle
역할 확인: req.user.role === 'consumer' 아니면 403 FORBIDDEN

입력 (body): qr_payload, consumer_lat, consumer_lng
  - 누락 시 400

처리 순서:
  1. consumer_id = req.user.id
  2. qr_payload 파싱: const [qr_id, sig] = qr_payload.split(':')
     파싱 실패(분리 불가) → 400 INVALID_QR
  3. merchant_qr_codes 조회 (qr_id)
     - 없으면 → 400 INVALID_QR + security_events 삽입
  4. status 확인:
     - 'completed' → 400 REPLAY_QR (실제로는 payment_sessions 기준이지만 단순화)
       ※ 명세 기준: status='pending' 아닌 active QR만 검증. QR이 used처리 후 revoked
       → status !== 'active' → REPLAY_QR or INVALID_QR (expired)
  5. HMAC 재계산 및 검증:
       hmac.verify(qr_id, merchant_id, amount, nonce, merchant_lat, merchant_lng, sig)
       ※ amount, nonce는 merchant_qr_codes에 없음 → 설계 재검토:
          spec 4.1의 서명 입력 = {payment_id}:{merchant_id}:{amount}:{nonce}:{lat}:{lng}
          merchant_qr_codes에는 amount/nonce 컬럼이 없음.
          → payment_sessions 테이블이 nonce/amount 보유.
          → QR 발급 시 amount/nonce도 merchant_qr_codes에 저장하거나,
            별도 payment_sessions INSERT 후 검증.
       실제 구현 방안:
          requestQR에서 nonce, amount도 merchant_qr_codes에 extra 컬럼으로 저장
          (스키마에 없으므로) → 또는 payment_sessions를 requestQR 시점에 생성.
          ★ 현재 스키마 제약: merchant_qr_codes에 amount/nonce 컬럼 없음.
            payment_sessions.nonce unique, qr_id FK.
          → 구현 결정: requestQR에서 payment_sessions도 함께 생성
            (consumer_id는 미정이므로 NULL 허용 필요 — 스키마 위반 가능성 있음)
            OR: verifyQR 시점에 payment_sessions 생성 (QR 자체에 amount 포함 안 됨)
          → 최종 결정: QR에 amount 미포함. QR은 "상점 신원 증명"만 수행.
            verifyQR body에 amount 포함 → 소비자가 금액 입력.
            서명 message = `${qr_id}:${merchant_id}:${merchant_lat.toFixed(4)}:${merchant_lng.toFixed(4)}`
            (amount/nonce 제외) — spec과 다르지만 현재 스키마와 정합
            ※ 이 결정은 PHASE 2 내에서 확정. 스키마 변경 없이 처리.
  6. HMAC 검증 실패 → 400 INVALID_QR + security_events 삽입
  7. Haversine 거리 계산:
       d = haversine.distance(merchant_lat, merchant_lng, consumer_lat, consumer_lng)
       d > MAX_DISTANCE_METERS → 400 LOCATION_MISMATCH + security_events 삽입
         detail: { payment_id: qr_id, merchant_lat, merchant_lng, consumer_lat, consumer_lng,
                   distance_m: d, max_allowed_m, message }
  8. consumer balance 조회:
       supabaseAdmin.from('users').select('balance, is_blocked').eq('id', consumer_id)
       is_blocked → 403 USER_BLOCKED
       balance < amount → 400 INSUFFICIENT_BALANCE
  9. 트랜잭션 (순서 중요, 원자성):
       a. consumers.balance -= amount
       b. merchant.balance += amount
       c. merchant_qr_codes.status = 'revoked' (또는 used 처리)
       d. transactions INSERT { qr_id, consumer_id, merchant_id, amount }
          ※ payment_request_id = qr_id (FK → merchant_qr_codes.qr_id)
       e. security_events INSERT { event_type: 'PAYMENT_OK', user_id: consumer_id,
            detail: { merchant_id, amount, distance_m: d } }
       ※ Supabase는 클라이언트 레벨 트랜잭션 미지원 → supabaseAdmin.rpc('pay_transaction', {...})
          또는 순서대로 실행 후 실패 시 rollback 시도 (베스트 에포트)
  10. 응답 200: { success: true, data: { message: '결제가 완료되었습니다.', amount, merchant_id } }

================================================================================
주의사항 및 결정 사항
================================================================================

1. HMAC 서명 입력에서 amount/nonce 제외 결정
   이유: merchant_qr_codes 스키마에 amount/nonce 컬럼 없음
   대안: QR = 상점 신원 증명 QR (amount는 소비자가 verifyQR body에 입력)
   서명: `${qr_id}:${merchant_id}:${merchant_lat.toFixed(4)}:${merchant_lng.toFixed(4)}`

2. 트랜잭션 원자성
   Supabase JS SDK는 클라이언트 트랜잭션 미지원
   → supabaseAdmin.rpc('pay_transaction', params) 으로 DB 함수 호출 권장
   → 또는 순서 실행 + 실패 보상 로직 (베스트 에포트)

3. SSE 방식
   Supabase Realtime 미사용 → 2초 폴링으로 구현
   마지막 전송 이벤트 id를 클로저에 보관

4. AI 분석 루프
   index.js 서버 시작 시 aiAnalyzer.startAnalysisLoop() 호출
   DISABLE_AI=true 이면 startAnalysisLoop()는 no-op

5. 소비자 좌표 저장 정책 (spec 4.4)
   consumer_lat/lng는 DB에 저장하지 않음
   LOCATION_MISMATCH 이벤트 detail에만 보안 감사 목적으로 기록
